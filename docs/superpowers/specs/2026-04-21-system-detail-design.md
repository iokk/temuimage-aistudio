# 电商出图工作台系统细节优化设计

- 日期: 2026-04-21
- 项目: 电商出图工作台 V15.2.1
- 设计主题: 全链路细节补强、信息架构收敛、危险操作防误触、文件与提供商治理
- 设计基调: 平衡型安全

## 1. 背景

当前系统已经具备本地个人工作流的主能力，但交互层存在几个明显问题：

1. 导航与功能操作混在一起，侧边栏承担了不该放在导航层的动作。
2. 任务、历史、文件、删除逻辑还没有形成统一的生命周期模型。
3. 危险操作缺少分级确认，容易造成“点了就删”或“点了像没反应”的体验。
4. 提供商设置虽然可用，但校验、连接测试、错误诊断和运行中保护仍然偏弱。
5. 页面成功、失败、禁用、重试、恢复等状态反馈不够一致，增加理解成本。

本设计的目标不是重写产品，而是在现有 Streamlit 单体结构之上做一次工程化整理，让系统更稳、更清楚、更适合持续迭代和回归测试。

## 2. 目标与非目标

### 2.1 目标

1. 让侧边栏重新回到“导航区”，不承载项目管理类操作。
2. 在项目中心统一承接任务、历史、回收站、文件管理。
3. 为删除、清理、切换、重试、保存等高风险操作补齐分级确认和状态反馈。
4. 建立项目文件生命周期，明确“历史删除”“移入回收站”“彻底删除”的区别。
5. 强化提供商设置，补齐校验、测试连接、错误分类、当前提供商保护。
6. 建立一套可执行的 bug 测试路径，覆盖回归、异常、边界条件。

### 2.2 非目标

1. 不把本轮设计扩展到多用户、团队协作、权限系统。
2. 不重构成多仓库或前后端分离架构。
3. 不在本轮设计中引入数据库替换现有 JSON 存储。
4. 不追求一次性做完所有视觉翻新，优先解决结构和状态安全问题。

## 3. 设计原则

1. 导航与操作分离: 用户先决定“去哪里”，再决定“做什么”。
2. 高风险动作可见、可理解、可撤销: 能回收的不直接删除，不能回收的强确认。
3. 状态统一: 禁用、加载、成功、失败、可重试都用一致的表达方式。
4. 文件优先透明: 每个动作都要说清楚会不会影响磁盘文件、ZIP、输入素材。
5. 配置先校验再保存: 错误尽量在提交前暴露，而不是运行时才炸。
6. 渐进式重构: 顺着当前页面与数据结构演进，不做脱离代码现实的大改。

## 4. 信息架构

### 4.1 侧边栏重构

侧边栏只保留以下内容：

1. 产品名与版本。
2. 当前提供商摘要。
3. 分层导航：功能区 / 管理与配置。
4. 极少量全局状态摘要，例如进行中任务数。

侧边栏移除以下内容：

1. 清理已完成任务。
2. 批量删除。
3. 打开文件夹。
4. 危险操作入口。

### 4.2 页面结构

建议将导航调整为双层结构：

左侧功能区：
1. `🚀 智能组图`
2. `🎨 快速出图`
3. `🏷️ 标题生成`

左侧管理与配置：
1. `🧩 模板库`
2. `⚙️ 提供商设置`
3. `🛠️ 系统设置`

右上角状态中心：
1. `📡 状态中心`
2. `📚 项目中心`

说明：

1. 现有“个人设置中心”拆为两个明确入口。
2. 模板库独立，统一承接标题模板、图片模板和后续模板资产。
3. 项目中心从侧边导航中弱化，转由右上角状态中心承接高频访问。
4. 提供商设置独立，避免被其他配置项稀释。
5. 系统设置只承接默认语言、默认模型、输出目录、自动清理策略、诊断项等系统级配置。

### 4.3 项目中心结构

项目中心使用顶部 Tabs：

1. `进行中`
2. `历史项目`
3. `回收站`
4. `文件管理`

推荐原因：

1. 单页内关联紧密，用户容易理解项目从“运行中”到“完成”再到“删除”的流转。
2. 比独立后台更轻，比单页混排更清楚。
3. 适合 Streamlit 当前结构，改造阻力小。

## 5. 项目生命周期模型

项目状态拆为两条线：

### 5.1 任务运行状态

1. `queued`
2. `running`
3. `done`
4. `error`
5. `cancelled`
6. `expired`

### 5.2 记录存储状态

1. `active_record`: 正常存在于历史项目
2. `trashed_record`: 已移入回收站，可恢复
3. `purged_record`: 已彻底删除，不可恢复

关键规则：

1. 任务终态不等于记录被删除。
2. “清理已完成”只作用于 `done` 类历史记录，不清理失败、取消、过期记录。
3. 普通删除默认进入回收站，而不是直接删文件。
4. 彻底删除才会删除项目目录、ZIP、输入素材副本和 manifest。

## 6. 项目中心交互设计

### 6.1 进行中

承载内容：

1. 任务名、任务类型、进度、开始时间、最近错误、使用提供商。
2. 操作：取消任务、查看日志、重试排队中的失败项。

规则：

1. 运行中的任务禁止删除。
2. 已取消任务保留到历史中，不在“进行中”显示。
3. 若任务使用的提供商已失效，显示风险标记，但不自动终止。

### 6.2 历史项目

承载内容：

1. 成功、失败、取消、过期项目的记录列表。
2. 预览图、标题内容、ZIP、输入素材计数、输出文件计数、时间、语言、提供商。

操作：

1. 打开文件夹
2. 下载 ZIP
3. 重新发起
4. 删除到回收站
5. 批量清理已完成

规则：

1. “清理已完成”按钮只出现在历史项目页顶部，不出现在侧边栏。
2. 该按钮仅在存在 `done` 记录时可点击。
3. 点击后展示确认层，明确显示：
   - 将清理的项目条数
   - 将移入回收站还是彻底删除
   - 是否影响本地项目文件
4. 成功反馈必须可读，例如“已将 12 条完成记录移入回收站，保留 3 条失败记录”。

### 6.3 回收站

承载内容：

1. 被删除的历史记录。
2. 删除时间、原项目名、文件数量、占用空间、过期时间。

操作：

1. 恢复
2. 彻底删除单条
3. 清空回收站

规则：

1. 恢复后回到历史项目，并恢复对应 manifest 关系。
2. 清空回收站使用强确认。
3. 可配置自动清空天数，例如 7 天、15 天、30 天、永不自动清空。

### 6.4 文件管理

承载内容：

1. 项目输出目录总览
2. 各项目目录空间占用
3. ZIP、图片结果、输入素材副本、错误日志计数
4. 不存在文件或路径异常的告警

操作：

1. 打开目录
2. 重新生成 ZIP
3. 清理孤儿文件
4. 导出诊断信息

规则：

1. 文件管理是项目中心的诊断视图，不直接替代 Finder。
2. 如果 manifest 中记录存在但磁盘文件缺失，需要显示“记录与文件不一致”的修复建议。

## 7. 危险操作分级

### 7.1 一级: 轻操作

例如：

1. 打开文件夹
2. 下载 ZIP
3. 切换 Tab

规则：

1. 不需要确认。

### 7.2 二级: 可恢复操作

例如：

1. 删除历史项目
2. 批量清理已完成
3. 删除提供商（非当前、且无任务绑定）

规则：

1. 需要确认。
2. 默认进入回收站或软删除。
3. 完成后给出撤销或恢复入口。

### 7.3 三级: 不可恢复操作

例如：

1. 清空回收站
2. 彻底删除项目文件
3. 删除当前默认提供商且无替代项

规则：

1. 强确认。
2. 明确后果。
3. 明确影响范围。

## 8. 按钮与状态反馈规范

### 8.1 按钮禁用规则

1. 无可用对象时禁用，不允许“点了没反应”。
2. 执行中的批量操作按钮进入 loading 态，避免重复提交。
3. 运行中任务相关危险操作禁用，并显示原因。
4. 当前提供商被任务引用时，删除按钮禁用或转为替换流程。

### 8.2 反馈文案规范

统一格式：

1. 成功反馈说清楚对象和数量。
2. 失败反馈说清楚原因和建议动作。
3. 空状态说清楚为何为空，以及下一步可做什么。

示例：

1. “没有可清理的已完成项目。”
2. “当前有 2 个任务正在使用该提供商，暂时不能删除。”
3. “已恢复 1 个项目，原文件夹和 ZIP 已重新关联。”

## 9. 提供商设置设计

### 9.1 页面结构

提供商设置页分为：

1. 当前提供商摘要
2. 提供商列表
3. 添加 / 编辑表单
4. 连接测试与诊断区
5. 风险提醒区

### 9.2 字段校验

保存前做同步校验：

1. 名称必填
2. API Key 必填
3. `gemini` 类型默认允许空 Base URL
4. `relay` 类型若为空 Base URL，给出警告或阻止保存
5. 模型名为空时提示将回退到全局默认
6. Base URL 格式不合法时阻止保存

### 9.3 提供商动作规则

1. 设为当前：若当前任务正在运行，不影响已运行任务，只影响后续新任务。
2. 删除提供商：若该提供商为当前默认，需先选择替代项或显式确认切空。
3. 禁用提供商：已运行任务不受影响，创建新任务时不可选。
4. 保存设置：只保存变更项，并反馈哪些字段已更新。

### 9.4 连接测试

连接测试分为三层：

1. 网络连通性
2. 鉴权有效性
3. 模型调用有效性

错误分类必须分开：

1. API Key 缺失或无效
2. Base URL 不可达
3. 代理错误
4. 账号或地区不支持
5. 模型名不可用
6. 上游配额不足

### 9.5 运行时保护

1. 不允许删除当前唯一可用提供商而不提示风险。
2. 不允许在没有可用提供商的情况下误点创建任务后才发现错误。
3. 当任务重发时，如果原提供商不存在，自动回退到当前可用提供商，并提示用户。

## 10. 系统设置设计

系统设置承接：

1. 默认标题语言
2. 默认图片文案语言
3. 默认模型
4. 默认标题模型
5. 默认视觉模型
6. 项目输出目录
7. 代理模式与代理地址
8. 自动清理策略
9. 回收站保留天数
10. 诊断与日志导出

新增建议：

1. 危险操作策略: 标准确认 / 极简确认
2. 文件保留策略: 永久保留 / 自动清理已完成项目输出副本
3. 调试模式: 显示详细错误、请求摘要、文件修复工具

## 11. 文件管理与存储设计

### 11.1 文件类型

每个项目目录应明确包含：

1. 输出图片
2. `titles.txt`
3. `errors.txt`
4. `download.zip`
5. `manifest.json`
6. `inputs/` 中的输入素材副本

### 11.2 删除策略

1. 删除到回收站: 记录迁移，文件保留。
2. 彻底删除: 记录与文件同步删除。
3. 孤儿文件: 文件存在但无记录；在文件管理中提供清理建议。
4. 孤儿记录: 记录存在但文件不存在；在历史项目中提示“文件已丢失”。

### 11.3 恢复策略

1. 回收站恢复应保留原项目名与原完成时间。
2. 若原目录缺失但 manifest 仍在，可尝试恢复记录并标记文件不完整。
3. ZIP 缺失时允许重新打包。

## 12. 错误处理设计

### 12.1 错误来源

1. 用户输入错误
2. 本地文件错误
3. 提供商配置错误
4. 网络或代理错误
5. 模型或上游服务错误
6. 内部状态不一致

### 12.2 错误表现形式

1. 表单级错误: 靠近字段显示
2. 操作级错误: 按钮附近显示
3. 页面级错误: 顶部警示区
4. 系统级错误: 诊断区和日志导出

### 12.3 恢复策略

1. 可重试错误提供“重试”入口
2. 不可重试错误提供“修改设置”或“切换提供商”入口
3. 文件类错误提供“打开目录”或“修复记录”入口

## 13. 测试设计

### 13.1 冒烟测试

1. 启动应用
2. 切换全部主页面
3. 添加并保存提供商
4. 创建任务
5. 查看任务进入项目中心
6. 完成后查看历史项目

### 13.2 高风险回归测试

1. 清理已完成项目
2. 删除历史项目
3. 回收站恢复
4. 清空回收站
5. 删除当前提供商
6. 切换默认提供商
7. 重发历史项目
8. 打开不存在的目录

### 13.3 异常测试

1. API Key 为空
2. Base URL 错误
3. 模型名错误
4. 输出目录不存在或无权限
5. ZIP 丢失
6. manifest 与文件不一致
7. 任务运行时切换提供商

### 13.4 UI 状态测试

1. 按钮在无对象时禁用
2. loading 态防止重复点击
3. 成功、失败、空状态文案符合规范
4. 批量操作反馈显示实际数量

## 15. 实现建议

为适配当前代码结构，建议先在 `app.py` 内做“局部模块化”而不是一次性拆大文件：

1. 抽出项目中心相关函数组
2. 抽出回收站与文件管理函数组
3. 抽出提供商校验与诊断函数组
4. 抽出统一确认与反馈 helper

这样可以在保持当前仓库可运行的前提下，逐步把复杂逻辑从页面渲染中分离出来。

## 16. 风险与决策

### 已确认决策

1. 安全与效率冲突时，采用平衡型安全。
2. “清理已完成任务”不应放在左侧导航区。
3. 整套设计范围包含任务、历史、文件管理、提供商设置、错误处理和测试修复。

### 当前新增决策

4. 模板系统按三阶段推进：
   - 阶段 1：真正的模板管理能力
   - 阶段 2：所见即所得编辑器
   - 阶段 3：更深层结构治理
5. 当前轮次只先完成阶段 1，不跨入阶段 2 和阶段 3 的能力开发。

### 当前风险

1. 当前项目不是 git 仓库，后续无法按常规提交设计与代码变更。
2. 现有 `app.py` 已较大，若不做局部模块化，后续修复会越来越难回归。
3. 现有任务数据为运行时内存存储，某些“清理任务队列”的行为在重启后语义有限，需要与历史记录逻辑区分清楚。

## 17. 本设计的完成定义

当以下条件满足时，认为本轮系统细节优化完成：

1. 侧边栏不再放置项目管理类危险操作。
2. 项目中心具备进行中、历史、回收站、文件管理四个清晰分区。
3. 所有删除和清理动作具备明确确认与结果反馈。
4. 提供商设置具备校验、测试连接和错误分类。
5. 文件记录与磁盘文件状态可被发现、解释和修复。
6. 有一套执行过的回归测试清单，并修复核心高风险 bug。
