# 电商出图工作台模板管理重构设计

- 日期: 2026-04-21
- 项目: 电商出图工作台 V15.2.1
- 主题: 系统设置中“图片模板管理”的信息架构重构
- 结论: 采用方案 A，按业务工作流分组

## 1. 背景

当前“系统设置 > 模板”中的图片模板管理存在几个明显问题：

1. 分组方式偏代码实现导向，使用的是 `combo_types` 和 `smart_types`，不符合用户心智。
2. 模板说明方式不清楚，用户很难快速理解“这个模板在哪个页面生效、解决什么问题”。
3. 业务分类与用途分类混在一起，导致“使用步骤”“主图白底”“对比图”等模板看起来像同一层级，但逻辑并不一致。
4. 当前页面是“上面一个维度、下面又一个维度”的展示方式，结构重复，理解成本高。

这类问题不会马上造成程序报错，但会直接削弱模板配置能力，使模板管理页像“开发配置页”，而不是“业务人员可理解的后台”。

## 2. 设计目标

本轮重构目标：

1. 让模板管理页按业务工作流组织，而不是按底层数据结构组织。
2. 让每个模板都能被用户快速回答三个问题：
   - 它在哪个页面会用到
   - 它主要负责什么类型的图片
   - 修改它会影响什么
3. 保持当前系统整体设计语言一致：
   - 主导航按业务入口组织
   - 项目中心按任务生命周期组织
   - 模板管理也应按业务工作流组织
4. 保留后续扩展空间，允许新工作流在未来继续增加模板分组。

## 3. 不采用的方向

### 3.1 不采用“纯用途分组”

例如只分为：

1. 展示类
2. 卖点类
3. 信息类
4. 场景类

原因：

1. 对当前系统来说过于抽象。
2. 用户首先想到的是“我在智能组图里改模板”，而不是“我要改一个展示类模板”。
3. 纯用途分组更适合模板量很大、工作流很多的系统，当前阶段会增加理解成本。

### 3.2 不采用“纯技术层级分组”

例如保留：

1. `combo_types`
2. `smart_types`

原因：

1. 这更像代码层概念，不像产品层概念。
2. 不利于后续继续做面向业务的后台收敛。

## 4. 最终方案

采用方案 A：按业务工作流分组。

模板管理页的图片模板部分，改为以下一级结构：

1. 智能组图模板
2. 快速出图模板
3. 翻译保版模板

说明：

1. 用户先按自己所在的工作页面找模板。
2. 每个模板分组内部，再展示该工作流下的具体模板项。
3. 标题模板继续单独管理，不与图片模板混排。

## 5. 新的信息架构

## 5.1 模板库中的模板页结构

建议结构如下：

1. 标题模板管理
2. 图片模板管理
   - 智能组图模板
   - 快速出图模板
   - 翻译保版模板

说明：

1. 模板资产应从“系统设置”中独立出去，成为一级 `模板库` 页面。
2. 标题模板和图片模板是两种不同产物，不应混成一个“大模板池”。
3. 图片模板内部再按页面场景分区，而不是按代码组名分区。

## 5.2 图片模板页内结构

每个工作流分组下，使用统一的模板卡片列表。

每张模板卡显示：

1. 模板名称
2. 模板用途说明
3. 当前适用页面
4. 是否启用
5. 排序
6. 模板提示语或 hint

模板卡不再只暴露技术字段，而是先让用户理解用途。

## 6. 模板命名重构

当前命名里有些条目过于技术化、内部化或层级不清。

建议统一命名风格：

1. 名称优先表达业务作用
2. 说明优先表达场景和输出目标
3. 避免“系统内部字段名式”的命名

### 6.1 智能组图模板建议

保留或微调为：

1. 主图白底
2. 功能卖点图
3. 场景应用图
4. 细节特写图
5. 尺寸规格图
6. 对比优势图
7. 包装清单图
8. 操作引导图

说明：

1. “使用步骤”建议改成“操作引导图”或“步骤说明图”。
2. 原因是“使用步骤”太像内容标签，不像模板类型。
3. 改名后，它能更自然地归入“智能组图工作流中的信息说明型图片”。

### 6.2 快速出图模板建议

保留但统一说明语言：

1. 卖点图
2. 场景图
3. 细节图
4. 对比图
5. 规格图

说明需要补清楚：

1. 它们是在“快速出图”页面中可选择的模板类型。
2. 适合快速生成，不强调复杂编排。

### 6.3 翻译保版模板建议

新增一个独立分组，而不是继续隐含在快速出图逻辑里。

建议只放与翻译保版相关的模板策略，而不是套用智能组图/快速出图模板。

例如：

1. 原图保版翻译
2. 合规替换翻译
3. 文案最小变更翻译

说明：

1. 翻译保版不是“图类型模板”，而是“处理策略模板”。
2. 把它独立出来，有助于后续把翻译工作流做完整。

## 7. 页面展示逻辑

### 7.1 展示顺序

推荐顺序：

1. 标题模板管理
2. 智能组图模板
3. 快速出图模板
4. 翻译保版模板

原因：

1. 与主导航中的业务入口一致。
2. 智能组图是更复杂、更核心的工作流，放在图片模板里优先展示。
3. 翻译保版作为相对独立的策略类模板，放在最后更清楚。

### 7.2 每组说明文案

每个工作流分组顶部都需要一句清晰说明。

例如：

1. 智能组图模板：
   - 用于 `智能组图` 页面中的可选组图类型，决定系统会生成哪些图片结构。
2. 快速出图模板：
   - 用于 `快速出图` 页面中的快捷模板选择，适合快速生成卖点图、场景图等标准类型。
3. 翻译保版模板：
   - 用于翻译保版类任务，控制原图保留程度与文案替换策略。

## 8. 数据结构建议

当前底层仍可保留现有 `combo_types` 和 `smart_types`，但产品层不再直接暴露这两个名字。

建议引入一层显示映射：

1. 工作流分组元数据
2. 每个模板的显示名称、展示说明、所属工作流
3. 页面上只渲染产品层标签

也就是说：

1. 代码可暂时保留旧结构，降低改造风险。
2. UI 层先完成“用户看得懂”的重构。
3. 后续再逐步把底层模板结构进一步归一化。

## 9. 迁移策略

推荐分两阶段推进。

### 阶段 1：UI 与命名重构

1. 不改模板实际功能逻辑。
2. 只改页面展示结构、分组标题、说明文案、名称。
3. 把“翻译保版”从描述层面独立出来。

### 阶段 2：模板模型重整

1. 为翻译保版建立独立模板数据组。
2. 把当前模板项补充更明确的用途元数据。
3. 让模板管理与实际页面选择逻辑完全对齐。

## 10. 用户体验要求

模板管理页应该做到：

1. 用户不用知道 `combo_types` 是什么。
2. 用户能从页面标题直接知道模板影响哪个工作页面。
3. 用户点开某个模板时，能立刻明白这个模板主要生成什么图。
4. 用户不会再困惑“为什么步骤图和主图白底属于同一组但没有解释”。

## 11. 风险与边界

### 风险

1. 如果只改名字不改说明，依然会产生新的歧义。
2. 如果把翻译保版硬塞进快速出图模板里，结构还会继续混乱。

### 边界

1. 阶段 1 先完成模板管理能力接入，不直接跳到复杂拖拽式编辑。
2. 阶段 2 采用轻量级所见即所得方案，先做实时预览，不做画布式布局。
3. 当前仍不引入模板版本管理。

## 12. 完成定义

当以下条件成立时，认为模板管理重构完成：

1. 页面上不再直接出现 `combo_types` / `smart_types` 这类技术分组概念。
2. 图片模板按业务工作流分为：
   - 智能组图模板
   - 快速出图模板
   - 翻译保版模板
3. “使用步骤”被改造成更清晰的业务命名，例如“操作引导图”。
4. 每个模板分组都带有明确说明文案。
5. 用户能够从模板页直接看懂“这个模板影响哪个页面、用来生成什么图”。

## 13. 阶段 1 执行边界

阶段 1 聚焦“真正的模板管理能力”。

阶段 1 允许实现：

1. 模板分组重构
2. 模板命名与说明重构
3. 翻译保版模板从只读说明升级为真实可编辑模板组
4. 模板修改接入实际翻译流程

阶段 1 不做：

1. 所见即所得模板编辑器
2. 拖拽式布局配置
3. 模板版本管理
4. 更深层的模板数据模型归一化

## 14. 后续阶段边界

### 阶段 2：所见即所得编辑器

当前已进入阶段 2 的第一版，实现范围如下：

1. 实时模板卡预览
2. 工作流模板列表预览
3. 名称、说明、启用、排序、Hint、Prompt 的边改边看
4. 翻译保版模板的 Prompt 预览型编辑

当前阶段 2 不做：

1. 拖拽布局
2. 画布式排版
3. 组件级自由编辑
4. 视觉样式像素级编辑

后续阶段 2 可继续进入：

1. 模板可视化编辑
2. 字段级预览
3. 页面级模板布局编辑
4. 更强的模板即时反馈

### 阶段 3：更深层结构治理

当前已开始第一轮：

1. 将模板管理渲染逻辑从系统设置主函数中抽离为独立 helper
2. 降低“设置页大函数”继续膨胀的风险
3. 为后续模板能力扩展和编辑器继续演进预留清晰边界
4. 模板保存前统一做规范化，减少空值、缺省值和排序异常带来的隐性耦合

后续再进入：

1. 模板底层数据模型统一
2. 模板与工作流配置彻底解耦
3. 结构混乱的统一收敛
4. 更完整的模板能力抽象

## 15. 当前执行进度

截至本轮，已完成：

1. 图片模板页按业务工作流重组为：
   - 智能组图模板
   - 快速出图模板
   - 翻译保版模板
2. 关键模板名称与说明完成业务化调整。
3. “使用步骤”升级为“操作引导图”。
4. 翻译保版模板已成为真实可编辑模板组。
5. 翻译保版模板已接入实际翻译任务流程。
6. 模板管理页已具备第一版所见即所得编辑能力，可实时预览模板卡与工作流中的展示效果。
7. 模板管理相关渲染逻辑已开始从系统设置主函数中拆分，进入更深层结构治理的第一轮。
8. 模板读写路径已统一增加规范化处理与 helper 收口，降低页面层直接依赖底层数据结构的风险。
