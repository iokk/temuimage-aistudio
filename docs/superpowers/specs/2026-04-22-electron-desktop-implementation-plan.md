# 电商出图工作台桌面化实施计划

- 日期: 2026-04-22
- 关联设计: `2026-04-22-electron-desktop-architecture-design.md`
- 实施目标: 基于已确认架构，将产品重构为 `Mac 优先`、`安装后即用`、`完整功能对齐` 的 Electron 桌面客户端
- 实施原则: 先打通主链路，再扩完整功能；每阶段均有可验证成果；优先避免返工

## 1. 范围确认

本实施计划以以下边界为前提：

1. 新客户端使用 `Electron + React/Vite + FastAPI + Python Worker + SQLite + 本地文件存储`。
2. 首版交付 `Mac`。
3. 首版要求安装后即可使用，不依赖用户自行安装 Python。
4. 功能范围完整覆盖现有主要功能面：
   - 智能组图
   - 快速出图
   - 图片翻译
   - 标题生成
   - 项目中心
   - 模板库
   - 提供商
   - 系统设置与诊断
5. 不兼容旧版本地数据。
6. 允许重构交互，不保留旧 Streamlit 页面结构。

## 2. 实施总策略

实施采用“主链路优先”的分阶段推进策略：

1. 先完成新工程骨架与运行时闭环。
2. 再完成公共底座能力。
3. 再打通一个最小完整业务闭环验证架构。
4. 然后平铺全部业务工作流。
5. 最后补齐管理能力、交付能力和内测修复。

核心原因：

1. 可以尽早验证最关键的桌面主链路。
2. 能在工作流铺开前发现架构问题。
3. 有利于在完整范围不变的前提下控制返工。

## 3. 里程碑定义

项目按 4 个里程碑管理：

### M1：桌面骨架跑通

完成标准：

1. Electron 可启动。
2. Renderer 可加载。
3. Python/FastAPI 可由 Electron 拉起。
4. SQLite 可初始化。
5. 前端可调用本地 API 获得系统状态。
6. Python 启动失败时有明确启动错误页。

### M2：首个完整工作流跑通

完成标准：

1. Provider 可配置并安全保存 Key。
2. 标题生成可提交任务。
3. 任务中心可看到进度与结果。
4. 项目中心可看到归档记录。
5. 错误能被统一展示并记录日志。

### M3：创作中心完整

完成标准：

1. 智能组图、快速出图、图片翻译、标题生成全部迁移完成。
2. 四类任务均接入统一任务系统。
3. 结果归档、文件落盘、项目详情查看均可用。

### M4：完整桌面产品可分发

完成标准：

1. 项目中心、模板库、设置、诊断全部可用。
2. 打包产物可在目标 Mac 设备安装与启动。
3. 应用具备启动页、错误页、日志目录与诊断能力。
4. 完成首轮内测修复，可进入更广泛分发准备。

## 4. 工作流拆分

实施拆分为 7 个工作流，每个工作流独立产出并在阶段中穿插推进。

### 4.1 工作流 A：工程与运行时

包含：

1. Electron Forge 工程初始化
2. React/Vite/TS 工程初始化
3. Python/FastAPI 工程初始化
4. SQLite 与 migration 基础设施
5. App 数据目录初始化
6. Python runtime 打包策略
7. 启动页与健康检查

### 4.2 工作流 B：桌面宿主能力

包含：

1. Preload API 设计
2. 文件选择
3. 目录选择
4. 打开路径
5. 获取版本与运行状态
6. 日志目录入口
7. 安全密钥读写

### 4.3 工作流 C：公共业务底座

包含：

1. Provider 数据模型
2. Settings 数据模型
3. Template 数据模型
4. Project/Task 数据模型
5. 通用错误模型
6. 通用 API client
7. 通用任务状态机

### 4.4 工作流 D：创作工作流

包含：

1. 标题生成
2. 图片翻译
3. 快速出图
4. 智能组图

### 4.5 工作流 E：管理与资产

包含：

1. 项目中心
2. 回收站
3. 文件管理
4. ZIP 重建
5. 模板库

### 4.6 工作流 F：系统能力

包含：

1. Provider 测试连接
2. 诊断页
3. 文件一致性检查
4. 模板健康检查
5. 日志聚合
6. 诊断包导出

### 4.7 工作流 G：交付与内测

包含：

1. DMG / APP 打包
2. 首启流程优化
3. 错误页与恢复流程
4. 内测清单
5. 稳定性修复
6. Mac 签名 / 公证准备

## 5. 分阶段执行计划

## 阶段 0：冻结旧系统、建立迁移清单

目标：

1. 停止继续扩大旧 Streamlit 结构负担。
2. 为新架构建立迁移映射。

任务：

1. 梳理现有功能模块及入口。
2. 梳理现有核心业务逻辑。
3. 梳理现有模板、设置、Provider、项目、任务相关数据项。
4. 标出“可复用核心逻辑”和“需要重写的 UI/状态逻辑”。
5. 建立新旧模块映射表。

交付物：

1. 功能映射清单
2. 数据映射清单
3. 核心逻辑迁移清单

验收标准：

1. 所有现有功能都能被映射到新客户端的对应模块。
2. 不存在“后面再补看”的隐性大模块。

## 阶段 1：新骨架落地

目标：

1. 跑通桌面应用的最小主链路。

任务：

1. 初始化 Electron Forge 工程。
2. 初始化 Renderer 工程并接入 React/Vite/TypeScript。
3. 初始化 Python/FastAPI 工程。
4. 建立 App 数据目录初始化逻辑。
5. 建立 SQLite 连接与 migration 基础设施。
6. 建立 Electron 启动 Python 进程逻辑。
7. 建立 Python 健康检查接口。
8. 建立启动状态页与启动失败页。
9. 建立基础日志输出。

交付物：

1. 可启动桌面应用骨架
2. 可启动的本地 FastAPI 服务
3. `GET /api/v1/system/health` 或等效系统状态接口

验收标准：

1. 打开客户端后可看到新前端界面。
2. Electron 能拉起 Python 服务并通过健康检查。
3. 前端能显示系统状态。
4. Python 服务无法启动时，用户能看到可读错误页。

依赖关系：

1. 无

## 阶段 2：公共底座能力

目标：

1. 先补齐后续所有工作流会依赖的公共能力。

任务：

1. 定义数据库 schema：
   - providers
   - settings
   - template_groups
   - templates
   - projects
   - project_files
   - tasks
   - task_events
2. 落地 repository 层。
3. 落地统一错误模型与 API 返回协议。
4. 落地 Preload API：
   - 选择文件
   - 选择目录
   - 打开路径
   - 获取版本
   - 获取运行时状态
5. 落地安全密钥存储适配层。
6. 落地 Provider、Settings、Templates 的基本 CRUD API。
7. 落地前端 API client 和基础状态管理。
8. 落地公共页面骨架和主导航。

交付物：

1. Provider 页面基础可用
2. Settings 页面基础可用
3. Template 数据结构可存取
4. 桌面文件能力可调用

验收标准：

1. 可以新建 Provider 并安全保存 Key。
2. 可以保存默认设置。
3. 可以在前端正确读写模板记录。
4. 可以通过桌面能力选择文件和打开路径。

依赖关系：

1. 依赖阶段 1 的工程骨架

## 阶段 3：打通首个完整工作流

目标：

1. 用一个最小但完整的业务流程验证整套架构。

优先工作流：

1. `标题生成`

任务：

1. 定义标题生成请求/响应模型。
2. 迁移标题生成核心业务逻辑到 service / worker。
3. 建立标题生成任务创建接口。
4. 建立标题生成任务执行器。
5. 建立结果归档逻辑：
   - 项目记录
   - 输出文件
   - 错误记录
6. 建立标题生成 UI：
   - 输入模式切换
   - 模板选择
   - 语言选择
   - 任务提交
   - 结果显示
7. 将其接入项目中心和任务中心。

交付物：

1. 可用的标题生成完整链路

验收标准：

1. 用户可配置 Provider 后完成标题生成。
2. 能看到任务创建、运行、完成或失败状态。
3. 结果能进入项目中心。
4. 错误信息可读、日志可追踪。

依赖关系：

1. 依赖阶段 2 的 Provider、Settings、Tasks、Projects 基础设施

## 阶段 4：扩展全部创作工作流

目标：

1. 完整迁移四大业务工作流。

推荐顺序：

1. 图片翻译
2. 快速出图
3. 智能组图

每个工作流统一遵循以下落地模板：

1. 定义请求模型
2. 定义任务类型
3. 建立 service 编排
4. 建立 worker 执行器
5. 建立文件归档与项目记录
6. 建立前端表单与结果展示
7. 建立任务取消、错误呈现与重新发起

交付物：

1. 四大创作流程全部可用

验收标准：

1. 智能组图可完成提交、生成、归档、查看。
2. 快速出图可完成提交、生成、归档、查看。
3. 图片翻译可完成提交、生成、归档、查看。
4. 标题生成仍保持稳定可用。
5. 四类任务共享统一任务中心和项目中心模型。

依赖关系：

1. 依赖阶段 3 对任务系统和项目归档链路的验证

## 阶段 5：管理与资产模块

目标：

1. 将完整产品补齐为可长期使用的桌面客户端。

任务：

1. 落地项目中心：
   - 进行中
   - 历史项目
   - 回收站
   - 文件管理
2. 落地项目筛选、搜索与详情视图。
3. 落地 ZIP 重建。
4. 落地回收站恢复与彻底删除。
5. 落地模板库：
   - 模板组
   - 模板列表
   - 模板编辑
   - 模板启用/排序
6. 落地批量操作与确认流程。

交付物：

1. 完整项目中心
2. 完整模板库

验收标准：

1. 用户能通过项目中心查看和管理全部历史任务。
2. 文件缺失、孤儿目录等问题可被感知。
3. 模板库可真正承担业务模板维护角色。

依赖关系：

1. 依赖阶段 4 已经产出足够项目数据供验证

## 阶段 6：系统诊断与交付打磨

目标：

1. 让应用从“功能可用”变成“可安装、可定位问题、可内测分发”。

任务：

1. 落地诊断页：
   - 运行环境
   - Provider 检查
   - 文件健康检查
   - 模板健康检查
2. 落地日志目录入口。
3. 落地诊断包导出。
4. 优化首启体验。
5. 优化错误恢复路径。
6. 建立 DMG / APP 打包产物。
7. 准备 Mac 签名与公证所需清单。
8. 进行首轮多设备内测并修复高优问题。

交付物：

1. 可分发安装包
2. 诊断能力
3. 内测修复记录

验收标准：

1. 客户端在目标 Mac 设备上能安装和启动。
2. 首次启动可进入工作区，不需要用户安装 Python。
3. 出现启动失败或任务异常时，用户有明确恢复路径。

依赖关系：

1. 依赖阶段 1-5 的功能闭环已经基本稳定

## 8. 每阶段 Definition of Done

每个阶段必须同时满足以下条件才算完成：

1. 功能可在新客户端中真正运行。
2. 有明确的错误展示路径。
3. 有日志记录。
4. 相关数据能进入 SQLite。
5. 相关文件能落入标准应用目录。
6. 前端不依赖临时 Mock 才能继续下一阶段。

## 10. 风险前置控制

实施过程中需要优先前置验证的风险点如下：

1. Electron 拉起内置 Python runtime 是否稳定。
2. 打包后资源路径解析是否稳定。
3. SQLite 与文件归档是否会出现不一致。
4. 任务并发限制是否足够稳妥。
5. 安全密钥存储链路是否可控。

建议这些风险不要等到最后阶段再验证，而是在阶段 1-3 就尽量落地。
